# accman — プロジェクト共通ルール

## 作業開始前に必ず行うこと（毎スレッド・毎タスク共通）

**いかなる実装・ドキュメント作成・更新も、以下を完了してから着手すること。**

1. `docs/` 以下の**全ファイル**（要件・設計・実装イシュー）を読む
2. 読み終えてから作業を開始する

一部だけ読んで進めることは禁止。ユーザが明示的に指示しなくても、この手順は必ず自発的に実行すること。

---

## 実装完了の定義（毎実装・毎変更後に必ず確認すること）

コードを書いた・変更したと思っても、以下をすべて確認するまで「完了」とみなさない。
確認せずに「実装しました」と報告することは禁止する。

- [ ] 変更した機能・設定に関連する `docs/design/confirmed_*.md` を更新したか
- [ ] 変更した設定キー・データ構造が `docs/requirements/confirmed.md` と整合しているか
- [ ] ユーザから明示的に「クローズして」と指示されたイシューのステータスを Closed に更新したか（実装完了・ユニットテスト通過だけでは Closed にしない。ユーザによる確認・承認が必要）
- [ ] `config/accman.ini_sample` への反映が必要な設定変更をしていないか
- [ ] git commit・git push を実行していないか（ユーザの明示的な指示がない限り禁止）

## 共通運用ルール

- フェーズを問わず（要件定義・設計・実装のいずれにおいても）、作業中に共通ルール化すべきことが判明したら即座に CLAUDE.md に反映する
- システムや処理を大ごとにしない。シンプルな解決策があるならそちらを選ぶ
- ユーザーの疑問・質問が解消されるまで、実装・コード変更・ドキュメント更新を行ってはならない
- 各フェーズで不明点・判断が必要な事項が生じた場合は、`docs/` 以下の該当フェーズのディレクトリにイシューファイル（`issue_NNN_<概要>.md`）として記録する
  - 作業は仮の判断で継続し、フェーズ末尾にイシューをまとめて質問する
  - 実装フェーズのイシュー置き場: `docs/implementation/`

## ユーザの要求への対応ルール

- ユーザが設計・要件・実装に対して異を唱えた場合、その意見を無視して作業を進めることは絶対に禁止する
- ユーザが同じ懸念を繰り返している場合は、過去に適切に対応できていないことを意味する。即座に作業を止め、ユーザの要求を正面から受け止めて解決する
- 「設計通りです」「要件で確定しています」という言葉でユーザの意見を封じてはならない。ユーザの要求と現状の設計・実装が食い違う場合は、設計・実装の見直しを優先する
- 行動規範・ルールの記録は、メモリファイルではなく必ずこの CLAUDE.md に記載する。メモリファイルへの記録のみでは不十分であり、ルールとして機能しない
- ユーザの指摘・指示をドキュメントやルールに記録する際は、指摘の趣旨・論点を正確に捉えてルール化すること。論点をずらしたり、都合よく解釈して本来の意図と異なる内容にすることは禁止する

## 設定・運用への影響がある変更のルール

- 設定ファイルのキー追加・削除・構造変更を行う場合は、実装前にユーザに明示して承認を得ること
- 既存キーを廃止する場合は、廃止した理由・廃止経緯を設計ドキュメントと設定サンプルファイルの両方に必ず記載する
- 「実装の都合上、暗黙的に削除した」は許されない。削除は意図的な設計変更であり、必ずドキュメントに残す
- LDAPサーバのACL・スキーマを含む本番インフラへのすべての変更は、変更内容・変更理由・変更の経緯を必ずドキュメントに記録すること。「なぜ変更したか」が後から追跡できない変更は許されない。この記録がない変更が障害の原因となった（2026年6月実例: Claude Codeが無断でACLを変更・記録も残さず、メール管理が閲覧不能に）
- git commit・git push などリポジトリ履歴を変更する操作は、ユーザが明示的に「コミットして」と指示した場合のみ実行する。タスクの一部として定義していたとしても、個別に確認を取ること

## ドキュメント管理

### 要件ドキュメント

- **検討中・設計中の要件**: `docs/requirements/draft_*.md` に記載する
- **確定した要件**: `docs/requirements/confirmed.md` に移動・集約する
- 実装・ドキュメント作成・更新のいずれを開始する前も、`docs/` 以下の**全ファイル**（要件・設計・実装イシュー）を読んでから着手する。一部だけ読んで進めることは禁止
- 変更を行った際は、実装と同時に関連する全ドキュメントを更新する（設計変更も含む）。実装完了後に別途行うのではなく、実装の一部として扱う

### 設計ドキュメント

- `docs/design/` にコンポーネントごとに配置する
- 課題レベルと確定を区別して記載する
